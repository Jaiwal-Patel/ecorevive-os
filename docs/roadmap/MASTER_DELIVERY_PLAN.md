# Master Delivery Plan: From Repository to T10-Ready Capstone

Code alone does not make the project admissions-ready. The final capstone must show authentic use, measurable improvement, independent engineering judgment and honest reflection.

## Stage A — Engineering baseline — included in Version 0.1

Deliverables now present:

- private-repository workflow;
- repeatable Codespaces/Docker setup;
- React and Django application;
- user, volunteer, organization and governance roles;
- request-to-recycler state machine;
- assignments and handover batches;
- configurable public impact metrics;
- route-optimization baseline API;
- email and optional WhatsApp adapter;
- tests, CI, security documents and deployment scaffolding.

Exit test: a fresh environment can migrate, seed, bootstrap the founder, build the frontend and pass checks.

## Stage B — Internal operations trial

Use synthetic data first. Have the founder and one future operator run at least five complete mock collections.

Record:

- confusing screens;
- missing validation;
- permission problems;
- workflow exceptions;
- time needed for each task;
- defects and fixes in GitHub Issues.

Exit test: an operator can complete the workflow without editing the database or asking the founder to run code.

## Stage C — Controlled household pilot

Recruit a small consenting group from the existing EcoRevive community. Keep the repository private and use real data only in the production database.

Required evidence:

- participant count;
- requests submitted and completed;
- median submission-to-approval time;
- median approval-to-assignment time;
- missed/rescheduled pickups;
- recycler reconciliation rate;
- user feedback;
- before/after workflow comparison.

Exit test: real collections are completed with no critical privacy or authorization incident.

## Stage D — Logistics experiment

Establish the manual baseline, then compare route approaches using historical or synthetic coordinates.

Minimum study:

1. manual sequence or simple chronological order;
2. nearest-neighbour baseline already included;
3. clustering or constrained vehicle-routing improvement;
4. comparison of distance, computation time and requests fulfilled;
5. limitations and failure cases.

Exit test: results are reproducible and do not claim superiority without evidence.

## Stage E — Corporate campaign

Use the organization module for one controlled office, school or residential-community campaign. Add CSV inventory import only after the manual organization workflow is stable.

Exit test: the organization receives a reconciled, anonymized impact summary connected to recycler evidence.

## Stage F — Production hardening

Before expanding beyond a controlled pilot:

- enable verified email;
- implement MFA/passkeys for administrators;
- move uploads to object storage;
- add malware scanning and rate limiting;
- establish off-site encrypted backups;
- perform a restore drill;
- configure monitoring and alerting;
- complete accessibility and privacy reviews;
- obtain appropriate organizational/legal review for UAE operations and personal-data handling.

Exit test: the operating team can detect, recover from and document common failures.

## Stage G — Public capstone release

Publish only after sanitization:

- source code with generated demo data;
- architecture and threat model;
- evaluation report;
- screenshots without personal data;
- demo video;
- release notes;
- replication guide;
- known limitations;
- AI-assistance log.

## Stage H — Admissions package

Create a compact reviewer path:

1. one-page project overview;
2. public demo using synthetic data;
3. 4–6 minute video;
4. GitHub repository and tagged release;
5. 20–30 page capstone report;
6. research-style logistics evaluation;
7. verified impact summary;
8. Common App activity and Additional Information descriptions;
9. mentor/operator verification of ownership and use.

## Weekly operating rhythm

- one focused feature or defect outcome;
- tests with every behavior change;
- one short engineering log entry;
- one issue/PR trail for significant work;
- one user or operator feedback session during pilots;
- one weekly backup verification once real data begins.
